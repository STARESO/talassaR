# talassaR

`talassaR` is an R preprocessing project used for the TALASSA cumulative-effects risk modelling work carried out for the **Parc naturel marin du Cap Corse et de l’Agriate (PNMCCA)**.

The repository contains scripts used to check, correct, harmonise, format and spatially aggregate heterogeneous marine-use and habitat datasets before their integration into the downstream REFC modelling workflow.

REFC stands for **risque d’effets cumulés**, or *risk of cumulative effects*. In this project, REFC is used as a spatial decision-support indicator combining:

- human activities
- the pressures potentially generated by these activities
- benthic habitats
- the sensitivity of habitats to each pressure
- spatial grids used as common analysis units

The repository is mainly focused on **data preparation**. It does not contain the full modelling project itself, but prepares the clean and structured inputs needed by the cumulative-effects risk model.

---

## Project context

The TALASSA study aims to map potential cumulative-effect risks linked to recreational activities and maritime traffic within the PNMCCA. The work is based on the integration of multiple spatial datasets describing maritime uses and benthic habitats.

The project focuses on activities such as:

- recreational boating
- anchoring
- mooring on buoys or dead bodies
- motorised and non-motorised nautical activities
- recreational fishing
- scuba diving and freediving
- passenger transport and maritime traffic
- AIS-derived vessel activity
- Donia anchoring records
- aerial survey observations

The preprocessing workflow transforms these datasets into harmonised and reusable spatial tables that can be crossed with activity-pressure matrices and habitat-pressure sensitivity matrices.

---

## What this repository is for

This repository is used to:

1. centralise file paths and project configuration
2. inspect and correct raw datasets
3. harmonise activity data using RESOBLO/OFB and TALASSA activity codes
4. harmonise habitat data using habitat typologies compatible with the sensitivity matrix
5. format matrices used by the REFC model
6. spatially aggregate activities and habitats into selected analysis grids
7. export final processed datasets to files or a PostgreSQL/PostGIS database
8. keep a reproducible record of the preprocessing workflow

---

## What this repository is not

This repository is **not** the complete REFC modelling project.

The downstream modelling scripts, Monte Carlo simulations, final REFC calculations and final cartographic production are handled in a separate modelling workflow. `talassaR` prepares the data required by that workflow.

## Folder description

|---|---|

| `main/` | Main production scripts. These scripts are the core preprocessing workflow and should be used for correction, formatting, gridding and export. |

## General workflow

The preprocessing workflow follows four main stages.

```text
Raw data
   ↓
Correction and quality checks
   ↓
Formatting to RESOBLO / TALASSA / NATHAB-compatible structures
   ↓
Spatial aggregation into analysis grids
   ↓
Final processed outputs for REFC modelling
```

In practice, the recommended workflow is:

1. configure paths in `config/paths.yml`
2. run correction scripts in `main/correction_*.R`
3. run formatting scripts in `main/formatage_*.R`
4. run gridding scripts in `main/carroyage_*.R`
5. export final outputs with `main/exports_bd.R`
6. use the final tables in the separate REFC modelling project.

## Requirements

The project is written in R and uses geospatial, data manipulation and database packages.

A typical setup requires:

- R
- RStudio or VSCode
- `renv`
- QGIS for some spatial inspection and manual geometry work
- PostgreSQL/PostGIS for database export and inspection
- access to the raw and processed TALASSA data folders.

Main R package families used by the workflow include:

- spatial processing: `sf`
- data manipulation: `dplyr`, `tidyr`, `purrr`, `stringr`
- configuration: `yaml`
- spreadsheets and tabular data: `readxl`, `openxlsx`, `readr`
- database connection: `DBI`, `RPostgres`
- dates and times: `lubridate`
- JSON / AIS data handling: `jsonlite`
- mapping and visual checks: `leaflet`.

The exact package versions should be restored through `renv`.

Clone the repository:

```bash
git clone https://github.com/STARESO/talassaR.git
cd talassaR
```

Open the R project:

```r
# In RStudio, open:
talassaR.Rproj
```

Restore the R environment:

```r
renv::restore()
```

Then edit the configuration file:

```text
config/paths.yml
```

This file must point to your local copies of the raw data, processed data, grids, matrices and outputs.

## Main scripts

The `main/` folder contains the production scripts used in the final preprocessing workflow.

### Correction scripts

| Script | Purpose |
|---|---|
| `correction_survols_check1.R` | First quality-control step for aerial survey data. Checks activity labels, navigation status, vessel categories and consistency with RESOBLO codes. Produces intermediate files that can be manually corrected if needed. |
| `correction_survols_check2.R` | Second quality-control step for aerial survey data after manual inspection or correction. Produces cleaner aerial survey data for formatting. |
| `correction_donia.R` | Checks and corrects Donia anchoring data. Filters and harmonises usable records, joins relevant activity codes and prepares Donia data for later formatting. |
| `correction_peche.R` | Checks and harmonises recreational fishing survey data. Used to prepare fishing observations before TALASSA formatting and gridding. |
| `correction_plongee.R` | Checks and harmonises recreational diving site data. Standardises names, categories and RESOBLO/OFB links before formatting. |
| `correction_ais.R` | Checks and corrects AIS data supplied for the project. Harmonises vessel categories, formats columns and prepares AIS pings for later TALASSA processing. |

### Formatting scripts

| Script | Purpose |
|---|---|
| `formatage_habitats.R` | Formats benthic habitat layers for TALASSA. Some habitat geometry corrections may be performed in QGIS before this script finalises the structure used by the model. |
| `formatage_activites.R` | Converts corrected activity datasets into TALASSA-compatible activity tables. This step links the corrected observatory/RESOBLO structure to the activity codes required by the REFC workflow. |
| `formatage_ais.R` | Converts corrected AIS pings into TALASSA-compatible AIS activity data. Uses activity codes defined during AIS correction. |
| `formatage_matrices.R` | Formats the activity-pressure and habitat-pressure sensitivity matrices into the structure required by the REFC modelling project. |

### Spatial aggregation scripts

| Script | Purpose |
|---|---|
| `carroyage_talassa_habitats.R` | Aggregates habitat data into the selected analysis grid. Produces habitat surface or proportion information by grid cell. |
| `carroyage_talassa_activites.R` | Aggregates TALASSA activity point data into the selected grid. Computes activity intensity indicators by grid cell and activity code. |
| `carroyage_talassa_ais.R` | Aggregates AIS data into the selected grid. Computes vessel movement state and distinguishes transit from stationary/anchoring situations before aggregation. |

### Export script

| Script | Purpose |
|---|---|
| `exports_bd.R` | Exports final processed TALASSA datasets to a dedicated PostgreSQL/PostGIS database. Useful for inspection, mapping and connection with QGIS or other downstream tools. |

## Function scripts

The `r/` folder contains reusable functions used by the main scripts.

| Script | Purpose |
|---|---|
| `fct_jointure_id.R` | Assigns grid cell identifiers to spatial activity or habitat data. This is a key step before aggregation by grid cell. |
| `fct_intensite_computation.R` | Computes activity intensity indicators by cell and activity. The function uses available intensity variables when possible rather than only counting points. |
| `fct_ais_reading.R` | Reads and compiles MarineTraffic AIS JSON files. Includes helper functions for converting AIS JSON data into tabular data or database-ready formats. |
| `fct_ais_grid.R` | Contains AIS-specific gridding functions. Computes vessel movement state, recodes AIS activities according to movement status and aggregates AIS records by grid cell. |
| `fct_category_map.R` | Creates quick exploratory maps for a selected category variable, mainly used for checking Donia or activity datasets visually. |
| `fct_map_peche.R` | Creates quick interactive maps for fishing survey data, mainly for visual checking and exploratory analysis. |

## Grids used in the project

The TALASSA workflow was designed to work with several spatial grids.

The main grids used in the project are:

- a rectangular ARP-type grid of 1 nautical mile, useful for a global view at the scale of the marine park
- a finer hexagonal grid of 1/5 nautical mile, useful for local interpretation and identification of priority areas.

Some scripts contain a parameter such as:

```r
choix_carroyage <- "hex5"
```

Check this parameter before running gridding scripts.

## Reproducibility notes

The repository is designed to improve reproducibility, but some important points must be kept in mind:

1. **Raw data are external**  
   The repository contains scripts, not the complete raw data archive. Local paths must be configured before running the workflow.

2. **Some spatial corrections are manual**  
   Some geometry corrections and visual checks may require QGIS.

3. **The order of scripts matters**  
   Formatting scripts generally require corrected data. Gridding scripts require formatted TALASSA data.

4. **Development scripts may not be production-ready**  
   Scripts in `dev/` are useful for understanding previous tests, but the final workflow should rely mainly on `main/`.

5. **The REFC model is separate**  
   This repository prepares input datasets but does not contain the full modelling and Monte Carlo simulation workflow.

## Suggested execution order

A typical complete run follows this order:

```text
1. Configure paths
   config/paths.yml

2. Correct source datasets
   main/correction_survols_check1.R
   main/correction_survols_check2.R
   main/correction_donia.R

   main/correction_plongee.R

3. Format corrected datasets
   main/formatage_habitats.R

   main/formatage_ais.R

4. Aggregate data into grids

   main/carroyage_talassa_ais.R

5. Export final datasets
   main/exports_bd.R
```

Before running the full workflow, it is recommended to run scripts step by step and inspect outputs after each major stage.

## Working with PostgreSQL/PostGIS

Some scripts can export or interact with a PostgreSQL/PostGIS database.

Before using database export scripts, check:

- the database host
- the database name
- user credentials
- target schema names
- table overwrite behaviour
- geometry column names and coordinate reference systems.

Database parameters should be adapted locally and should not be committed with private credentials.

---

## Working with QGIS

QGIS is used alongside R for:

- visual inspection of spatial layers
- checking suspicious geometries
- manual correction of some spatial layers
- cartographic inspection of processed outputs
- connection to PostgreSQL/PostGIS tables.

When geometry processing is too slow or difficult in R, some corrections may be performed in QGIS and then re-imported into the R workflow.

---

## Data caution

The final processed datasets are intended for cumulative-effects risk modelling and decision-support analysis. They should not be interpreted as direct measurements of ecological impact.

The REFC approach produces a relative spatial risk indicator based on available data, matrices, assumptions and spatial aggregation choices.
